## L1 — `closeTask` εκτός transaction + σιωπηλή απώλεια audit

**Αρχείο:** `lib/core/database/tasks_repository.dart:1196–1227`.

**Σενάριο:**
Σε αντίθεση με τα `createTask` / `updateTask` / `insertCall` / `cloneCall` (που τυλίγουν
`update` + `audit` σε ένα `db.transaction`), το `closeTask` κάνει:
1. `db.update(...)` (κύρια εγγραφή), και μετά **ξεχωριστά**
2. `AuditService.log(...)` μέσα σε `try { … } catch (_) {}`.

Δύο προβλήματα:
- **Μη ατομικό:** αν η καταγραφή audit αποτύχει, η αλλαγή κατάστασης έχει ήδη γραφτεί — μένει
  εκκρεμότητα «κλειστή» **χωρίς** εγγραφή audit. Το σφάλμα καταπίνεται σιωπηλά (`catch (_)`),
  οπότε ούτε καν φαίνεται.
- **Χωρίς έλεγχο ύπαρξης:** το `db.update` δεν ελέγχει το πλήθος επηρεασμένων γραμμών. Αν το
  `id` δεν υπάρχει (0 γραμμές), καταγράφεται ψευδώς «ΚΛΕΙΣΙΜΟ ΕΚΚΡΕΜΟΤΗΤΑΣ» για ανύπαρκτη
  εγγραφή. (Το `updateTask` το κάνει σωστά: `if (n > 0 && oldRow != null)`.)

**Προτεινόμενη διόρθωση:**
Τύλιξε update+audit σε `db.transaction`, όπως το `updateTask`. Έλεγξε `n > 0` πριν το audit.
Μη καταπίνεις το σφάλμα σιωπηλά — άσε το να κάνει rollback το transaction (ή τουλάχιστον
κατέγραψέ το). Ιδανικά, επαναχρησιμοποίησε το ίδιο μοτίβο diff (`oldRow`) για συνέπεια.

**Στοχευμένα τεστ:**
```
flutter test test/core/database/tasks_repository_test.dart
```
(Αν δεν υπάρχει: δημιούργησε τεστ που κλείνει task, επιβεβαιώνει status=closed **και** ύπαρξη
εγγραφής audit· και ότι κλείσιμο ανύπαρκτου id δεν παράγει audit.)

## L2 — `copyWith` δεν μπορεί να μηδενίσει πεδίο (Dart footgun)

**Αρχείο:** `lib/features/calls/models/equipment_model.dart:197–220` (και πιθανώς `Task`,
`CallModel` με ίδιο μοτίβο).

**Σενάριο (λανθάνον, όχι ενεργό bug σήμερα):**
Το `copyWith` χρησιμοποιεί `code: code ?? this.code`, `location: location ?? this.location`,
κ.λπ. Αυτό σημαίνει ότι **δεν γίνεται** να καθαρίσεις ένα πεδίο σε `null` μέσω `copyWith`:
`equipment.copyWith(location: null)` κρατά την παλιά τοποθεσία αντί να τη σβήσει. Έλεγξα ότι
**κανένας τρέχων καλών δεν περνά ρητό null**, οπότε δεν σπάει τίποτα τώρα — αλλά είναι κλασική
παγίδα: την επόμενη φορά που κάποιος θελήσει «καθάρισε το τμήμα/τοποθεσία» με copyWith, θα
βρει σιωπηλό bug.

**Προτεινόμενη διόρθωση (προαιρετική / προληπτική):**
Αν χρειαστεί μηδενισμός, χρησιμοποίησε το μοτίβο sentinel (π.χ. `Object? location = _sentinel`)
ή ξεχωριστές μεθόδους `clearLocation()`. Χαμηλή προτεραιότητα — άφησέ το ως γνωστό περιορισμό
αν δεν υπάρχει ανάγκη μηδενισμού.

**Τεστ:** δεν χρειάζεται αλλαγή τώρα· αν υλοποιηθεί sentinel, τεστ round-trip μηδενισμού.

## L3 — `updateCall` δεν εφαρμόζει defaults όπως το insert

**Αρχείο:** `lib/core/database/calls_repository.dart:182–201`.

**Σενάριο (οριακό):**
Το `_callInsertMap` βάζει defaults: `status ?? 'completed'`, `is_priority ?? 0`,
`lansweeper_state ?? 'unsent'`. Το `updateCall` όμως γράφει απευθείας `call.status`,
`call.lansweeperState` χωρίς fallback. Αν ένα `CallModel` φτάσει στο update με `status == null`
ή `lansweeperState == null`, γράφεται **NULL** στη στήλη — ασυνέπεια με τη ροή εισαγωγής. Το
`is_priority` προστατεύεται (`?? 0`), τα άλλα δύο όχι.

**Προτεινόμενη διόρθωση:**
Επιβεβαίωσε αν το `CallModel` εγγυάται non-null `status`/`lansweeperState`. Αν όχι, εφάρμοσε
τα ίδια defaults με το insert (`?? 'completed'`, `?? 'unsent'`) στο `updateCall`.

**Στοχευμένα τεστ:**
```
flutter test test/core/database/calls_repository_test.dart
```

## P1 — N+1 + επαναφόρτωση ρυθμίσεων σε `createTasksForOrphanCalls`

**Αρχείο:** `lib/core/database/tasks_repository.dart:1257–1299`.

**Σενάριο:**
Ο βρόχος για κάθε ορφανή κλήση:
- κάνει **ξεχωριστό** `db.query('users', …)` ανά ορφανή (γραμμή 1265),
- καλεί `createFromCall(...)` που εσωτερικά (γραμμή 309) κάνει
  `await getTaskSettingsConfig()` — δηλαδή **ξαναδιαβάζει τις ρυθμίσεις από τη βάση για ΚΑΘΕ
  κλήση**, και ανοίγει **ξεχωριστό transaction** ανά κλήση.

Για 100 ορφανές κλήσεις: ~100 queries χρηστών + 100 αναγνώσεις ρυθμίσεων + 100 transactions.
Σε μαζική δημιουργία αυτό είναι αισθητά αργό.

**Προτεινόμενη διόρθωση:**
- Φόρτωσε το `TaskSettingsConfig` **μία φορά** πριν τον βρόχο και πέρασέ το (το
  `buildCreateFromCallRow` ήδη δέχεται `config`· απλώς χρειάζεται overload/παράμετρος στο
  `createFromCall` ή απευθείας χρήση του `createFromCallOnExecutor`).
- Κάνε **ένα** query χρηστών με `WHERE id IN (...)` για όλα τα `callerId`.
- Τύλιξε όλες τις εισαγωγές σε **ένα** transaction (χρήση `createFromCallOnExecutor(txn, …)`).

**Στοχευμένα τεστ:**
```
flutter test test/core/database/tasks_repository_test.dart
```
(Επιβεβαίωσε ίδιο αποτέλεσμα: N ορφανές → N tasks, με σωστά ονόματα/ημερομηνίες.)

## P2 — `getHistoryCalls` χωρίς `LIMIT`

**Αρχείο:** `lib/core/database/calls_repository.dart:390–453`.

**Σενάριο:**
Το ερώτημα ιστορικού δεν έχει `LIMIT`/`OFFSET` — επιστρέφει **όλες** τις κλήσεις που ταιριάζουν
στα φίλτρα, με βαρύ JOIN (υποερώτημα `GROUP_CONCAT` τηλεφώνων ανά χρήστη + 5 LEFT JOIN). Σε βάση
με δεκάδες χιλιάδες κλήσεις, φορτώνεται όλο το σύνολο στη μνήμη σε κάθε άνοιγμα/φιλτράρισμα
ιστορικού. Υπάρχει ήδη ξεχωριστό `getHistoryCallCount`, που δείχνει ότι η σελιδοποίηση ήταν
πρόθεση αλλά δεν εφαρμόστηκε στο κύριο query.

**Προτεινόμενη διόρθωση:**
Πρόσθεσε προαιρετικά `limit`/`offset` (παράμετροι, με `?`) και σελιδοποίηση στο UI ιστορικού
(lazy scroll ή σελίδες). Εναλλακτικά ένα λογικό όριο (π.χ. 500 πιο πρόσφατες) με «φόρτωση
περισσότερων».

**Στοχευμένα τεστ:**
```
flutter test test/core/database/calls_repository_test.dart
```

## P3 — Query μέσα σε loop στα sparklines

**Αρχείο:** `lib/core/database/tasks_repository.dart:915–939` (μέσα στο `_buildSparklines`).

**Σενάριο:**
Για κάθε ημέρα του παραθύρου εκτελείται ξεχωριστό `db.rawQuery` για τα overdue (γραμμή 928).
Σήμερα το sparkline είναι σταθερά 7 ημερών, άρα μόνο 7 queries — μικρή επίπτωση. Το μοτίβο
όμως είναι «N queries για N ημέρες» και θα κλιμακωθεί άσχημα αν κάποτε μεγαλώσει το παράθυρο.

**Προτεινόμενη διόρθωση (χαμηλή προτεραιότητα):**
Αντικατέστησε με μία ομαδοποιημένη query ή υπολόγισε τα overdue από τα ήδη φορτωμένα δεδομένα
(created/due ανά ημέρα) εν μνήμη, όπως γίνεται ήδη για active/closed/cancelled.

**Στοχευμένα τεστ:**
```
flutter test test/features/tasks/
```

## P4 — `setState` σε κάθε πλήκτρο ξαναχτίζει όλη την ενότητα

**Αρχείο:** `lib/features/directory/screens/widgets/equipment_form_remote_params.dart:328–331`.

**Σενάριο:**
Στο `onChanged` κάθε πεδίου παραμέτρου καλείται `setState(() {})`, που ξαναχτίζει ολόκληρο το
`_buildRemoteParamsSection` (dropdown ζώνης Α + όλα τα πεδία + επανυπολογισμός `vncDefault`
κ.λπ.) σε **κάθε πάτημα πλήκτρου**. Για μικρή φόρμα είναι ανεκτό, αλλά είναι περιττό rebuild.

**Προτεινόμενη διόρθωση (χαμηλή προτεραιότητα):**
Απομόνωσε το κομμάτι που όντως εξαρτάται από την τιμή (π.χ. η προειδοποίηση «χρειάζεται
παράμετρο») σε μικρότερο widget, ή ενημέρωσε το `_remoteParamValues` χωρίς πλήρες `setState`
όταν δεν αλλάζει η ορατότητα. Μη το κυνηγήσεις αν δεν φαίνεται lag.

**Τεστ:** δεν απαιτείται (καθαρά UI perf).

## M1 — Λανθασμένος ελληνικός πληθυντικός σε μηνύματα πλήθους

**Σημεία (επιβεβαιωμένα, αφύλακτα ως προς count=1):**

  «Μαζική επεξεργασία (**N εξοπλισμός**)»: πάντα **ενικός**, ενώ N>1 (π.χ. «3 εξοπλισμός»).
- `lib/features/settings/screens/remote_tools_management_screen.dart:111, 171, 301` —
  «σε **N εξοπλισμούς**»: πάντα **πληθυντικός**, οπότε «1 εξοπλισμούς» (σωστό: «1 εξοπλισμό»).
- `lib/features/directory/screens/widgets/categories_tab.dart:168` —
  «για **N κατηγορίες**»: για N=1 → «1 κατηγορίες».
- `lib/features/directory/screens/widgets/departments_tab.dart:245` —
  «για **N τμήματα**»: για N=1 → «1 τμήματα».

**Σενάριο:** όλα προσπελάσιμα με επιλογή 1 στοιχείου (ή N στο bulk), άρα εμφανίζουν αντιγραμματικό
μήνυμα. Καθαρά αισθητικό.

**Προτεινόμενη διόρθωση:**
Πρόσθεσε έναν μικρό βοηθό, π.χ. `String ελληνικόςΠληθυντικός(int n, String ενικός, String
πληθυντικός) => n == 1 ? ενικός : πληθυντικός;` και χρησιμοποίησέ τον στα παραπάνω σημεία
(«εξοπλισμός/εξοπλισμοί», «εξοπλισμό/εξοπλισμούς», «κατηγορία/κατηγορίες», «τμήμα/τμήματα»).

**Στοχευμένα τεστ:** unit-τεστ στον βοηθό πληθυντικού (n=0,1,2).

## M2 — `LIMIT $limit` με interpolation αντί για `?`

**Αρχείο:** `lib/core/database/tasks_repository.dart:524` (`getUpcomingTasks`).

**Σενάριο:** το `LIMIT $limit` περνά με string interpolation. Λειτουργικά είναι ΟΚ (int),
αλλά είναι ασυνεπές με π.χ. `getRecentCalls` (calls_repository.dart) που περνά το `LIMIT ?`
ως παράμετρο. Καθαρά θέμα συνέπειας/στυλ.

**Προτεινόμενη διόρθωση:** χρησιμοποίησε `LIMIT ?` με το `limit` στα `whereArgs`.

**Τεστ:** καλύπτεται από υπάρχοντα tasks_repository tests.

## M3 — SnackBar μετά από `pop()` με context διαλόγου (παρατήρηση)

**Αρχείο:** `lib/features/calls/screens/widgets/quick_call_dialog.dart:103–110`.

**Σενάριο (χαμηλή βεβαιότητα):**
Στην επιτυχία, το `_submit` κάνει `Navigator.of(context).pop()` και **αμέσως μετά**
`ScaffoldMessenger.maybeOf(context)?.showSnackBar(...)` με το context του διαλόγου που μόλις
έκλεισε. Συνήθως δουλεύει (το `maybeOf` βρίσκει τον messenger της κύριας οθόνης πριν
απο-προσαρτηθεί το element), αλλά είναι εύθραυστο μοτίβο «lookup μετά από pop». Τα σφάλματα
αντίθετα χρησιμοποιούν `showDialogSnackBar` (εντός διαλόγου) — σκόπιμο, αφού ο διάλογος μένει
ανοιχτός.

**Προτεινόμενη διόρθωση (προαιρετική):**
Κράτησε αναφορά στον `ScaffoldMessengerState` της κύριας οθόνης **πριν** το `pop`, ή δείξε το
success SnackBar από τον καλούντα (π.χ. αυτόν που άνοιξε τον διάλογο) μετά την ολοκλήρωση.

**Τεστ:** widget test που υποβάλλει γρήγορη κλήση και επιβεβαιώνει pop + μήνυμα.
