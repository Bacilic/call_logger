# Βάση Δεδωμένων

Εκαθάριση του lib\core\database\database_helper.dart μετά την ολοκλήρωση της εφαρμογής. Περιέχει πολλές εκδόσεις βάσης και πολλά άχριστα IF αφού η δομή της βάσης σταθεροποιήθηκε

 - Εύρεση σφαλμάτων: Ονόματα χωρίς τμήματα, ορφανά τηλέφωνα, κλπ

# Διωρθώσεις
- Να προστεθεί λειτουργία μετακίνησης εικόνας κατά την σχεδίαση του ορόφου. Στο ισόγιο δεν μπορώ να δω όλο τον όροφο και πρέπει να τον μετακινήσω ώστε να εστιάσω πχ στα μαγειρια

- Σμύκρινση γραμματοσειράς στο όνομα του τμήμματος στο χάρτη. Χρήσιμο όταν είναι πολά μαζί στρημογμένα (πχ στα γραφεία)

- το διπλό κλικ στην όχι πρώτη λέξη στο πεδίο τηλέφωνο στη επεξργασία χρήστη οδηγέι πάντα στην επιλογή της πρώτης λέξης. Μόνο στο πεδίο σημειώσεις συμπεριφέρεται σωστά το διπλό κλικ

### Άξονας 1 — Διαρροές Κατάστασης (State Leaks)

```
Αναλύεις τους Riverpod providers της εφαρμογής call_logger για πιθανές διαρροές κατάστασης.

1. Εξέτασε κάθε αρχείο provider που περιέχει εγγραφή στη βάση δεδομένων (insert/update/delete).
   Τα αρχεία που σε ενδιαφέρουν βρίσκονται στα:
   @call_entry_provider.dart @smart_entity_selector_provider.dart @call_header_provider.dart
   @tasks_provider.dart @task_service_provider.dart
   @department_directory_provider.dart @equipment_directory_provider.dart
   @category_directory_provider.dart @directory_provider.dart

2. Για κάθε StateNotifier / AsyncNotifier που εκτελεί mutation:
   - Καταγράφεις ποιους άλλους providers επηρεάζει λογικά η αλλαγή αυτή.
   - Ελέγχεις αν μετά την εγγραφή υπάρχει ρητή κλήση ref.invalidate / ref.refresh
     στους εξαρτημένους providers.
   - Σημειώνεις περιπτώσεις όπου ο εξαρτημένος provider δεν ενημερώνεται
     και το UI μπορεί να εμφανίζει ξεπερασμένα δεδομένα.

3. Ελέγχεις επίσης:
   - Providers που χρησιμοποιούν ref.watch σε άλλον provider αλλά δεν
     αντιδρούν όταν αυτός αλλάζει (π.χ. επειδή η εξάρτηση είναι σε
     παλιό snapshot και όχι στο reactive stream).
   - AsyncNotifier που κάνουν early return χωρίς να ενημερώνουν το state
     σε περίπτωση σφάλματος.

4. Παρέχεις έξοδο αυστηρά στη μορφή:
   [Αρχείο] → [Μέθοδος/Provider] → [Πρόβλημα] → [Πιθανή επίπτωση στο UI]

Οι παραπάνω οδηγίες λειτουργούν ως υπόδειξη — κάνε τους δικούς σου ελέγχους στον κώδικα για παραλήψεις και σφάλματα.
```

### Άξονας 2 — Συμβάσεις Βάσης (SQL Contract Violations)

```
Αναλύεις τη βάση δεδομένων και τα repository αρχεία της εφαρμογής call_logger
για παραβιάσεις συμβάσεων SQL και λανθασμένη διαχείριση δεδομένων.

1. Σάρωσε όλα τα αρχεία ΕΚΤΟΣ του φακέλου core/database/ για άμεσες
   κλήσεις σε DatabaseHelper ή sqflite API (π.χ. db.query, db.insert,
   db.rawQuery). Αναφέρεις κάθε εύρημα ως παραβίαση του κανόνα
   «SQL μόνο στα Repositories».
   Αρχεία ελέγχου: @calls_screen.dart @tasks_screen.dart @directory_screen.dart
   @history_screen.dart @lamp_screen.dart @lamp_migration_service.dart

2. Στα Repository αρχεία ελέγχεις:
   @calls_repository.dart @directory_repository.dart @settings_repository.dart
   @remote_tools_repository.dart @dictionary_repository.dart

   Για κάθε SELECT ερώτημα:
   - Φιλτράρεται ρητά η στήλη is_deleted = 0 όπου απαιτείται;
   - Φιλτράρεται ρητά η στήλη lansweeper_state όπου η λογική το απαιτεί;
   - Στήλες που επιτρέπουν NULL (π.χ. caller_id, equipment_id, department_id)
     — χειρίζεται το Dart model σωστά την περίπτωση null χωρίς να κρασάρει;

3. Ελέγχεις τη συνέπεια μεταξύ σχήματος και model:
   Συγκρίνεις τις στήλες του πίνακα calls (όπως ορίζονται στο @database_v1_schema.dart)
   με τα πεδία του @call_model.dart — υπάρχουν στήλες που υπάρχουν στο σχήμα
   αλλά δεν αντιστοιχούν σε πεδίο model ή αντίστροφα;
   Επαναλαμβάνεις για tasks / @task.dart και equipment / @equipment_model.dart

4. Παρέχεις έξοδο αυστηρά στη μορφή:
   [Αρχείο] → [Μέθοδος/Ερώτημα] → [Πρόβλημα] → [Πιθανή επίπτωση]

Οι παραπάνω οδηγίες λειτουργούν ως υπόδειξη — κάνε τους δικούς σου ελέγχους στον κώδικα για παραλήψεις και σφάλματα.
```

### Άξονας 3 — Ασύγχρονες Συνθήκες Ανταγωνισμού (Async Race Conditions)

```
Αναλύεις τον ασύγχρονο κώδικα της εφαρμογής call_logger για συνθήκες
ανταγωνισμού και μη ασφαλείς async μεθόδους.

1. Εξέτασε τα παρακάτω αρχεία που περιέχουν σύνθετη async λογική:
   @call_entry_provider.dart @smart_entity_selector_provider.dart
   @task_service.dart @database_backup_service.dart
   @lamp_migration_service.dart @lansweeper_sync_service.dart
   @database_maintenance_service.dart

2. Για κάθε async μέθοδο που τροποποιεί κατάσταση μετά από await:
   - Υπάρχει guard μεταβλητή (π.χ. _isLoading, _isSaving) που εμποδίζει
     διπλή ταυτόχρονη εκτέλεση;
   - Αν η μέθοδος καλείται δύο φορές γρήγορα (π.χ. διπλό tap σε κουμπί),
     τι συμβαίνει στην κατάσταση;
   - Μετά από κάθε await, ελέγχεται αν το context/ref είναι ακόμα έγκυρο
     πριν την ανάθεση νέου state;

3. Αναζήτησε συγκεκριμένα αυτά τα patterns:
   - setState() ή state = ... μετά από await χωρίς mounted check σε StatefulWidget.
   - Future.wait() ή παράλληλα awaits που τροποποιούν τον ίδιο πίνακα SQLite.
   - Timer ή Future.delayed που δεν ακυρώνονται στο dispose().
   - showDialog / Navigator push μετά από await χωρίς context.mounted check.

4. Στο κλείσιμο παραθύρου (AppShortcuts, window close flow):
   @app_shortcuts.dart @database_exit_backup.dart @database_helper.dart
   Ελέγχεις αν υπάρχει πιθανότητα η ροή κλεισίματος να εκτελεστεί δύο φορές
   ταυτόχρονα ή να διακοπεί στη μέση λόγω exception χωρίς να κλείσει
   η σύνδεση με τη βάση.

5. Παρέχεις έξοδο αυστηρά στη μορφή:
   [Αρχείο] → [Μέθοδος] → [Σενάριο ανταγωνισμού] → [Πιθανή επίπτωση]

Οι παραπάνω οδηγίες λειτουργούν ως υπόδειξη — κάνε τους δικούς σου ελέγχους στον κώδικα για παραλήψεις και σφάλματα.
```

### Άξονας 5 — Κενά Επαγγελματικής Λογικής (Business Logic Gaps)

```
Αναλύεις την επαγγελματική λογική (business logic) της εφαρμογής call_logger
για σιωπηλές αποτυχίες και μη χειριζόμενες οριακές περιπτώσεις.

1. Ροή δημιουργίας/αποθήκευσης κλήσης:
   @call_entry_provider.dart @call_header_provider.dart @calls_repository.dart
   Ακολούθησε τη ροή από το «Αποθήκευση» κουμπί μέχρι την εγγραφή στη βάση.
   - Τι γίνεται αν ο χρήστης πατήσει «Αποθήκευση» ενώ δεν έχει επιλεγεί
     καλών (caller_id = null) και caller_text είναι κενό;
   - Τι γίνεται αν η κατηγορία δεν υπάρχει πλέον στη βάση (διαγράφηκε
     εν τω μεταξύ);
   - Αν η αποθήκευση αποτύχει (exception), ενημερώνεται ο χρήστης ή
     χάνεται σιωπηλά η κλήση;

2. Ροή διαχείρισης εκκρεμοτήτων (tasks):
   @task_service.dart @tasks_provider.dart @task_form_dialog.dart
   - Τι γίνεται αν μια εκκρεμότητα συνδέεται με κλήση που έχει διαγραφεί
     (is_deleted = 1); Εμφανίζεται στη λίστα; Μπορεί να κλείσει κανονικά;
   - Το snooze — αν η ώρα του συστήματος είναι λανθασμένη ή αλλάξει
     (π.χ. θερινή ώρα), τι συμβαίνει με τις προθεσμίες;
   - Αν δύο χρήστες στο κοινό δίκτυο κλείσουν την ίδια εκκρεμότητα
     ταυτόχρονα, τι καταγράφεται στο audit;

3. Ροή μεταφοράς LAMP:
   @lamp_migration_service.dart @lamp_transfer_wizard_dialog.dart
   @old_equipment_repository.dart
   - Αν μια οντότητα (π.χ. χρήστης) υπάρχει ήδη στην κύρια βάση με το
     ίδιο όνομα, τι γίνεται; Δημιουργείται διπλότυπο ή απορρίπτεται;
   - Αν η μεταφορά διακοπεί στη μέση (π.χ. κλείσιμο παραθύρου), η βάση
     παραμένει σε συνεπή κατάσταση; Υπάρχει transaction rollback;
   - Καταγράφονται οι μεταφερθείσες οντότητες στο audit_log;

4. Ροή backup και συντήρησης βάσης:
   @database_backup_service.dart @database_maintenance_service.dart
   @database_exit_backup.dart
   - Αν ο χώρος του δίσκου εξαντληθεί κατά το backup, τι γίνεται με
     το ημιτελές αρχείο; Καθαρίζεται;
   - Αν το backup on exit αποτύχει, εμποδίζεται το κλείσιμο της εφαρμογής
     ή αγνοείται το σφάλμα;
   - Το retention (διαγραφή παλιών backup) — αν αποτύχει, επηρεάζεται
     το κυρίως backup;

5. Παρέχεις έξοδο αυστηρά στη μορφή:
   [Αρχείο] → [Ροή/Σενάριο] → [Οριακή περίπτωση] → [Τι συμβαίνει σήμερα] → [Πιθανή επίπτωση]

Οι παραπάνω οδηγίες λειτουργούν ως υπόδειξη — κάνε τους δικούς σου ελέγχους στον κώδικα για παραλήψεις και σφάλματα.
```
